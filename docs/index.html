Dependencias bajo control, en tu máquina

zscan consulta OSV sobre npm, PyPI, Go, RubyGems y Maven. Un solo scan genera un informe unificado Markdown/JSON: dependencias, árbol, pistas de imports, scraping por defecto de URLs de advisories (extractos en caché; usa --no-enrich-docs para ir más rápido) y prompt-scan según zscan.yaml — Markdown, agentes tipo AGENTS.md y código fuente donde haya literales tipo prompt (TS, Python, Go, …). zscan init descubre globs al crear el YAML. LLM opcional (config, llm-probe). No modifica tu repositorio.

Características

Multi-ecosistema + OSV

Un solo comando puede mezclar lockfiles presentes en la raíz del proyecto y consultar vulnerabilidades conocidas.

Árbol y propagación

Árbol resuelto para npm; vista plana para otros ecosistemas (MVP). Notas de impacto en cadena en el informe.

Imports y submódulos

Pistas best-effort por lenguaje; opción para ignorar rutas bajo submódulos Git al mapear imports.

Informe único scan

Un comando mezcla lockfiles presentes, OSV, enriquecimiento de referencias (por defecto) y evaluación de prompts[]. scan --no-prompt-llm evita llamar al modelo en ese bloque.

Prompts e init

zscan init propone globs según lo que exista: docs, prompts, .cursor, AGENTS.md, README, código TS/JS/Python/Ruby/Go/Java/Kotlin con literales largos tipo LLM. Reglas YAML + heurísticas + LLM si llm.enabled.

Offline OSV

Caché local de consultas OSV para entornos sin red (scan --offline o ZSCAN_OFFLINE=1).

Referencias CVE en el informe

Por defecto se descargan URLs de references, HTML → texto y extractos en Markdown/JSON (meta.docSnippets, caché ~7 días). Omite con --no-enrich-docs. La flag --enrich-docs quedó obsoleta (compat).

Servidor HTTP + Ollama

zscan serve expone POST /scan con el mismo núcleo. En desarrollo: npm run ollama:ensure y tests de integración locales con ZSCAN_INTEGRATION_OLLAMA=1.

VS Code, pre-commit

Tareas de ejemplo en contrib/vscode/; hook de muestra en contrib/pre-commit/. Plantilla de variables: .env.example.

Instalación desde el código

Requiere Node.js 18+. Clona el repositorio y compila en local (no hace falta publicar en npm para usar la herramienta). 

git clone https://github.com/angieshadai/zscan.git
cd zscan
npm install
npm run build

Opcional: en la raíz del repo, chmod +x zscan y usa ./zscan scan (el script instala y compila si hace falta).

Comandos rápidos

node dist/cli.js init --root .
node dist/cli.js scan --root . --markdown reporte.md
node dist/cli.js scan --root . --no-enrich-docs --markdown reporte.md   # sin scraping CVE
node dist/cli.js scan --root . --no-prompt-llm --markdown reporte.md   # prompts sin LLM
node dist/cli.js config --root .
node dist/cli.js llm-probe --root .
node dist/cli.js prompt-scan --root .
node dist/cli.js serve --port 8787

# Atajo en el repo clonado (instala/compila si hace falta):
./zscan scan --markdown reporte.md

# Desarrollo del propio zscan
npm test
npm run test:integration:ollama

Referencia completa (variables ZSCAN_*, CLI, VS Code, pre-commit): README → Referencia de configuración.

Referencias CVE: scraping en el informe

OSV aporta metadatos y enlaces; el texto de la página del advisory aporta contexto para remediación. En cada scan (salvo que uses --no-enrich-docs) zscan descarga esas URLs, reduce HTML a texto y añade extractos al Markdown y al JSON (meta.docSnippets), con caché en disco (TTL ~7 días). No sustituye a OSV: complementa el informe. 

# Comportamiento por defecto (con scraping):
node dist/cli.js scan --root . --markdown reporte.md

# Sin red a esas URLs / más rápido:
node dist/cli.js scan --root . --no-enrich-docs --markdown reporte.md

Variable ZSCAN_ENRICH_CACHE_DIR. API POST /scan: omite scraping con "enrichDocs": false. Roadmap en vision-agente-seguridad.md (§3).

Esta web con GitHub Pages

Este sitio es HTML estático en la carpeta docs/. Para publicarlo sin acciones ni npm:

  1. Repositorio en GitHub → Settings → Pages.
  2. Build and deployment: origen Deploy from a branch.
  3. Rama main (o la predeterminada) y carpeta /docs.
  4. Guardar. La URL será https://<usuario>.github.io/<repo>/.

El archivo docs/.nojekyll evita que GitHub procese el sitio con Jekyll y garantiza que los archivos se sirvan tal cual.
